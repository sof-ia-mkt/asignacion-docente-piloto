Regla de oro. Antes de cualquier operación riesgosa —recarga masiva de datos, cambios de esquema, pruebas que escriben en la base, o una demo importante— haz un respaldo primero. Toma segundos y es tu seguro contra "se rompió la plataforma".

Qué está protegido y dónde vive

PiezaDónde se respaldaCómo se recupera
Código (la app)GitHub (repo privado)Clonar / revertir un commit
Esquema (estructura de tablas)db/migrations/ en el repoCorrer las migraciones en orden
Datos (docentes, grupos, asignaciones…)backups/respaldo_*.json.gz (local, NO en GitHub)restaurar_db.mjs

Los datos contienen información personal de docentes (PII), por eso los respaldos no se suben a GitHub (están en .gitignore). Viven en tu Mac, en la carpeta backups/ dentro del proyecto.

Cómo hacer un respaldo

Desde la carpeta del proyecto (/Users/eme/asignacion-docente-piloto):

# Crea un respaldo nuevo con fecha y hora automáticas
node scripts/respaldo_db.mjs

Genera un archivo como backups/respaldo_2026-06-03-16-40-00.json.gz (comprimido) y te imprime cuántas filas guardó por tabla. Seguro Solo lee la base, nunca la modifica.

Snapshot consistente. El respaldo lee todas las tablas dentro de una sola transacción repeatable read read only: aunque alguien use la plataforma mientras corre, el respaldo captura una foto coherente de un mismo instante.

Cómo restaurar (recuperar la base)

Operación destructiva. Restaurar borra TODO el contenido actual de las tablas y lo reemplaza por el del respaldo. Recuerda que la base es compartida entre local y producción: restaurar afecta a las dos. Úsalo solo cuando de verdad quieras volver a un punto anterior.

Paso 1 — Ensayo (no escribe nada)

node scripts/restaurar_db.mjs backups/respaldo_2026-06-03-16-40-00.json.gz

Sin --confirmar es solo un ensayo: te muestra qué contiene el respaldo y el orden seguro de inserción, pero no toca la base. Seguro

Paso 2 — Restaurar de verdad

node scripts/restaurar_db.mjs backups/respaldo_2026-06-03-16-40-00.json.gz --confirmar

Destructivo Borra y recarga todo dentro de una sola transacción: si algo falla a mitad, no se aplica nada y la base queda intacta. Al terminar reajusta las secuencias para que los próximos registros no choquen.

Buenas prácticas

  • Respalda antes de cada demo y antes de cargar/normalizar datos masivamente.
  • Guarda una copia de los .json.gz más importantes fuera de la Mac (un disco externo o tu Drive personal). Si la laptop falla, no pierdes los respaldos.
  • No borres los respaldos viejos a la ligera; pesan poco (~120 KB).
  • Nombra/anota qué pasó antes de cada respaldo importante (ej. "antes de cargar grados sept").

Red de seguridad adicional: respaldos automáticos de Supabase

Además del respaldo local, Supabase (donde vive la base) ofrece respaldos administrados:

  • En el panel de Supabase: Database → Backups. Respaldos diarios automáticos.
  • Con el plan Pro se habilita PITR (Point-In-Time Recovery): volver a cualquier minuto.
  • Esto cubre el caso de que tu Mac no esté disponible. Recomendado para producción.
